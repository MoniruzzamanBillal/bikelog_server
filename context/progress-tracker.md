## Recent Activity

- **Review of specs 06–07** (`maintenanceType`, `engineOilType` — already implemented, per the user's request to check them). No bugs found in either module: `createXIntoDB`/`getXFromDB`, models, validation, and routes all match their specs exactly. This also resolved the one open question spec 06 itself flagged ("this duplicate-key catch may be redundant, confirm during implementation") — traced through `globalErrorHandler.ts`: the codebase-wide generic `error?.code === 11000` handler (`handleDuplicateError.ts`) returns **400** with the raw Mongo error string as the top-level `message` (not 409, not a clean message). So the manual try/catch in both services is **not redundant** — it's what upgrades a duplicate-name conflict from a leaky 400 to a clean `409 "...already exists"`. Confirmed via `yarn build` (clean) and `npx eslint` on both modules + `src/scripts` (0 errors, only expected `no-console` warnings in the seed scripts). No code changes needed.
  - **Flagged, not fixed (out of scope for 06/07):** the same generic 400-with-raw-message path is still what handles a duplicate **email** on `POST /auth/register` (`user.services.ts`'s `createUser` has no manual 11000 catch, unlike maintenanceType/engineOilType) — registering with an existing email currently returns a 400 with a raw `E11000 duplicate key error...` string as the message, not a clean 409. Same root cause, different module (spec 02, already implemented) — worth a follow-up if the user wants it addressed.

- **Bug-fix review of specs 03–05** (`bike`, `fuelLog`, `mileageRecord` — already implemented, per the user's request to "go through their implementation" and fix any issues found). Found and fixed:
  - `Bike` had no immutable "starting" odometer — only the denormalized `currentOdometer`, which every fuel/maintenance log bumps. Added a new `initialOdometer` field (snapshot of `currentOdometer` at creation, never changed after), set in `bike.service.ts`'s `createBikeIntoDB` and stripped from `updateBikeInDB`'s payload like `owner`/`currentOdometer`.
  - `fuelLog.service.ts`'s mileage-closure fallback (no prior full-tank fill) was reading `bike.currentOdometer` **after** `bumpOdometerIfHigher` had already mutated it to equal the new fill's own reading — this collapsed `distanceKm` to 0 for every bike's very first `MileageRecord`. Fixed to anchor on the new immutable `bike.initialOdometer` instead.
  - Same file's period-liters query used `$gte: periodStartDate` instead of `$gt`, double-counting the previous closing full-tank fill's own liters into the next period's `litersConsumed` (understating every mileage figure after the bike's first period). Fixed to `$gt`.
  - `mileageRecord.service.ts`: `computeMileageForRange`'s no-prior-log fallback had the same `currentOdometer`-vs-`initialOdometer` bug as above (wrong for a bike's first month/year of data once any later log had bumped `currentOdometer`) — fixed. `getLifetimeMileageFromDB` anchored on the earliest fuel log's reading rather than the bike's true starting point, undercounting any km ridden before the first-ever fuel log — fixed to use `initialOdometer`.
  - `getMileageRecordsFromDB`'s "approximate" rolling-average was computed by averaging existing `MileageRecord`s — which never exist for a user who never does a full-tank fill, making the fallback permanently non-functional for exactly the case spec 05/the plan doc built it for. Rewrote to compute the rolling average from the trailing 10 raw `FuelLog`s (regardless of `isFullTank`), matching spec 05; response shape changed from `{records, approximateMileage}` to `{exactRecords, approximate: {mileageKmPerLiter, basedOnFuelLogCount, isEstimate} | null}`.
  - `validateRequest.ts` called `Schema.parseAsync()` but discarded the result instead of reassigning it to `req.body` — meaning zod's default unknown-key-stripping never actually reached the controller/service layer, so extra client-supplied fields (e.g. a spoofed `isDeleted` or `owner` in a request body) passed through unstripped. Fixed to assign `req.body = parsed.body`.
  - Verified: `yarn build` clean, `npx eslint` clean on all touched files (`bike`, `fuelLog`, `mileageRecord`, `validateRequest.ts`).
